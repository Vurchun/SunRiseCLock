STM32 Timer PWM

STM32 / 01/03/2023 12/03/2023 / bare-metal, c, pwm, registers, stm32, timer

The Timer peripheral of microcontrollers is a versatile tool that finds various applications in generating Pulse-Width Modulated (PWM) signals. These signals serve to control and drive many external devices, from motor speed controllers and telecommunication systems to voltage regulation and more. In this article, we will explore the concept of PWM signals and demonstrate how to configure a timer peripheral for PWM signal generation in STM32 microcontrollers using bare-metal development techniques. By the end of this article, readers will have a thorough understanding of how to generate and manipulate PWM signals in their projects.

The Pulse-Width Modulated Signal

In electronics and communication, modulation is manipulating one or more properties of a carrier signal to transmit information through the signal medium. Although there are many modulation types and properties that can be modulated, we will focus on features related to PWM.

The carrier signal is a periodic signal with a fixed base frequency and this frequency also applies to the signal frequency of the PWM signal as PWM modulation does not alter with the frequency. Also, the PWM modulation is applied over the digital signals, meaning that the amplitude of the modulated signal has two voltage levels, namely high and low. In general, MCUs utilized 3.3 V as a high voltage level and 0 V (or GND) as a low voltage level.

Duty Cycle

The modulated signal has a fixed frequency, but the width of the pulses changes depending on the information carried. If the information has a higher value, the pulse width is higher, and vice versa. The width of a pulse can range from 0% of the signal period to 100% of the signal period (some systems may utilize a different range, e.g., servo motors utilize the range of 5% to 10%). The pulse width is generally called the “duty cycle” in the context of PWM. Therefore, a 10% duty cycle PWM signal’s voltage is high for 10% of the signal period and low for 90% of the signal period.

The basic idea behind PWM is to vary the width of a series of pulses while keeping the frequency of the pulses constant. The width of the pulses is varied in proportion to the information that is being transmitted. For example, if the signal being transmitted represents an audio waveform, the width of the pulses can be varied in proportion to the amplitude of the audio signal.

The PWM signal is typically generated using a microcontroller or dedicated PWM module. The microcontroller or module sends a train of pulses, with a fixed frequency and variable width, to the output pin of the device. The width of the pulses is controlled by a duty cycle, which is the percentage of time the pulse is on in each cycle. A high percentage duty cycle corresponds to a wide pulse, while a low duty cycle corresponds to a narrow pulse.

Timer Peripheral PWM Feature

In the previous two articles we have utilized the STM32F410’s TIM6 peripheral but the TIM6 does not support PWM generation. To generate a PWM signal we need to utilize either a general-purpose or an advanced timer. We will select the TIM11 peripheral due to its relatively simple register structure.

The general-purpose and advanced timer peripherals also come with one (or more) configurable channel(s). These channels can be utilized for capture/compare operations, one-pulse output, and PWM signal generation. The TIM11 peripheral supports one independent channel.

The TIM11 also supports all the features TIM6 offers. PWM signal is generated depending on the Capture/Compare Register’s value (CCR1). Also, we need to configure the proper channel and pin to support this feature as well.

When utilizing the PWM feature the PSC and ARR define the frequency of the signal together with the clock source frequency of the timer. The CCRx defines the duty cycle of the PWM signal.

Detailed Register Descriptions

Pulse Width Modulation mode allows generating a signal with a frequency determined by the value of the TIMx_PSC and TIMx_ARR registers, and a duty cycle determined by the value of the TIMx_CCRx register.

The CCR1 register’s value is required to be greater than 0 and less than or equal to the ARR’s value. Otherwise, the PWM signal will not have a percentage duty cycle. When the timer is enabled and configured for PWM signal generation the counter value increases (or decreases if it is counting down). And the counter value resets when it reaches the ARR value. The output signal will toggle when the CNT value reaches the CCR1 value. And it turns back to its initial state when the counter value resets.

In the figure below, we can observe that if the CCRx is 4, the signal changes its level (also triggers an interrupt) when the counter value reaches 4. When the counter overflows, the signal returns to its initial state. A similar situation can be observed if the CCRx value is 8. If the CCRx value is greater and the ARRx value, the signal stays at its initial state. When the ARRx is zero, it stays at the other signal level.

We can write any value between 0 and 65,535 in the CCR1 but to generate PWM signal it needs to be 0 < CCR1 ≤ ARR. If the CCR1 is 0 the duty cycle is 0% and if it is greater than the ARR value, the duty cycle is 100% (this situation can be reversed depending on the signal polarity, etc. that we will discuss later).

There are two PWM modes that can be configured, PWM mode 1, and 2.

  • PWM mode 1 : Channel 1 is active as long as TIMx_CNT < TIMx_CCR1, else inactive.
  • PWM mode 2 : Channel 1 is inactive as long as TIMx_CNT < TIMx_CCR1, else active.

And the PWM mode is configured by configuring the OC1M bitfield under the Capture/Compare Mode Register CCMR1. To configure for PWM mode 1 the bitfield is set to 6 (0b110), and to configure for PWM mode 2 the bitfield is set to 7 (0b111).

Also, it is a good practice to enable the preload of the output compare register. Meaning that the value written in the CCR1 will take effect after an update event. Otherwise, if the CCR1 value is changed the new value takes effect before finishing the current PWM signal period. To enable the preload of CCR1 we can set the Output Compare Preload Enable (OC1PE) bit of the CCMR1 register.

We can also configure what it means for the channel to be active or inactive under the Capture/Compare Enable Register CCER. If we set the bit CC1P to 0 the OC1 will be active high. For 1 the output will be active low.

These registers on top of the registers we have discussed in the two previous articles will cover the majority of the configuration that needs to be done over the timer peripherals. Now we are ready to move on to a demonstration to test the PWM mode.

Demo

In this demo, we will utilize the PWM to change the brightness level of an LED. To achieve our goal we will configure the system clock, and the timer peripheral. And we will build our application after this configuration. We will also utilize the timer interrupts to change the duty cycle of the PWM signal periodically. This will create a “breathing” effect on the LED. By utilizing the interrupts we can run more important tasks in the main loop if we need to. Now let’s jump right in!

Configuring The System Clock

The system clock configuration is the same as the previous articles. We have discussed this configuration in a previous article in detail.

Configuring The Timer

We will configure the TIM11 peripheral for PWM signal generation. After the configuration, the PWM signal will be generated without even calling the NVIC interrupt handler. Therefore this method requires no runtime to be spent to generate the PWM signal. Later we will utilize the interrupt handler to change the duty cycle of the signal.

To configure the TIM11 we can follow these steps:

  1. Enable peripheral clock sources (GPIOx, and TIMx clock sources)
  2. Set the mode of the GPIO pin, and set the alternate function
  3. Set the counter mode
  4. Set the timing parameters
  5. Set PWM mode and enable output compare preload register
  6. Enable auto-reload preload register
  7. Configure signal polarity, and enable the CC output
  8. Set interrupt source
  9. Load the register data to the hardware (preload–>shadow)
  10. Configure and enable interrupt
  11. Set the NVIC to handle interrupts

1. Enabling The Clock Sources

TIM11’s channel is PC12 by default. We will enable both the TIM11 and the GPIOC peripherals’ clock sources. The TIM11 is connected to the APB2 and the GPIOC is connected to the AHB1. We will set the related registers’ bitfields under the RCC peripheral.

RCC->APB2ENR |= RCC_APB2ENR_TIM11EN;    // Enable TIM11 peripheral clock source
RCC->AHB1ENR |= RCC_AHB1ENR_GPIOCEN;    // Enable GPIOC peripheral clock source

Unless the clock source of the peripherals is enabled we cannot reach and configure the peripherals’ registers.

2. GPIO Setup

First, we will configure the GPIOC’s pin #12 as ‘alternate function’ as we will be utilizing it with another peripheral. Then we will set the speed of the pin to high speed. The high speed’s allowed frequency is more than enough for us. Then we will select the proper alternate function for the pin. To find the correct alternate function we need to check the datasheet of the MCU.

In the datasheet, the alternate function mapping for each GPIO pin is provided. Under the TIM11 column, we can observe that the TIM11-CH1 is provided for the PC12. And we can observe that the TIM11 requires the pin to be configured as AF3.

The alternate function configurations are separated into two parts namely Alternate Function Register Low (AFRL) and Alternate Function Register High (AFRH). And they are provided under the CMSIS library as AFR[0] and AFR[1] respectively. The low register is responsible for pins #0 to #7, and the high register is for #8 to #15. Therefore our pin’s AF configuration is under the AFR[1].

We will set the bitfield starting in the 16th position to 3 for AF3.

GPIOC->MODER |= ( 0x2UL << GPIO_MODER_MODER12_Pos );		// PC12 Alternate function
GPIOC->OSPEEDR |= ( 0x2UL << GPIO_OSPEEDR_OSPEED12_Pos );	// PC12 High speed
GPIOC->AFR[1] |= ( 0x3UL << 16);				// AF3 for TIM11

3. Counter Mode

As the TIM11 peripheral only supports counting up there is nothing to be configured.

4. Timer Timing Parameters

Then we will set the registers of the TIM11 peripheral responsible for the PWM signal’s frequency and the duty cycle. I have parameterized these values.

TIM11->PSC = psc - 1;     // Set prescaler
TIM11->ARR = arr;         // Set auto-reload
TIM11->CCR1 = arr / 2;    // 50% duty cycle

5. Configure PWM Mode

We will set the CC mode to PWM mode 1, meaning that channel 1 is active as long as CNT < CCR1, else inactive. Also, we will enable the preload register of the CCR1.

TIM11->CCMR1 |= ( 0x6UL << TIM_CCMR1_OC1M_Pos );    // PWM mode 1
TIM11->CCMR1 |= TIM_CCMR1_OC1PE;		    // Enable preload of CCR1

6. Enable Auto Reload Preload

We will also enable the preload register of the ARR.

TIM11->CR1 |= TIM_CR1_ARPE;    // Enable preload of ARR

7. Signal Polarity, CC Output

Then, under the CCER we will set the signal polarity as active high by writing 0 to the CCP1 bit. And, we will enable the Capture Compare by setting the CC1E bit.

TIM11->CCER &= ~TIM_CCER_CC1P;     // Set signal polarity
TIM11->CCER |= TIM_CCER_CC1E;      // Enable CC

8. Set Interrupt Source

Now, we will set the interrupt source to prevent the UG bit from generating an interrupt. Therefore if we require to update the registers manually with UG the TIM11 won’t generate an interrupt.

TIM11->CR1 |= TIM_CR1_URS;    // Setting UG bit does not gen. int.

9. Reload The Registers

We will generate an update event by setting the UG bit. As a result of this update event, the values of the registers we have configured will be moved to the shadow registers of the TIM11 peripheral. Therefore, we ensure that the timer has loaded the correct values.

TIM11->EGR |= TIM_EGR_UG;    // Generate an update event

10. Configure and Enable Interrupt

Other than the Update Interrupt, we will also enable the Capture Compare 1 Interrupt of the TIM11 peripheral under the Interrupt Enable Register (DIER). The CC1I will be triggered when the CCR1’s content is equal to the CNT value of the timer. Both of the interrupt flags can be cleared by writing zero to their flag values under the Status Register (SR).

TIM11->DIER |= ( TIM_DIER_CC1IE | TIM_DIER_UIE );    // Update and CC interrupt enabled
TIM11->SR &= ~( TIM_SR_CC1IF | TIM_SR_UIF );         // Clear the interrupts

11. NVIC Configuration

We will configure the NVIC in a similar way we have done in the previous article. We will set the priority of the interrupt with the NVIC_EncodePriority() helper function. And, after that, we will enable the IRQn. Again we use the maximum priority but the priority levels must be considered carefully when dealing with multiple IRQns.

IRQn_Type IRQn = TIM1_TRG_COM_TIM11_IRQn;
uint32_t priorityGroup = NVIC_GetPriorityGrouping();
uint32_t preemptPriority = 0;
uint32_t subPriority = 0;
NVIC_SetPriority ( IRQn, NVIC_EncodePriority(priorityGroup, preemptPriority, subPriority) );
NVIC_EnableIRQ ( IRQn );

Wrapping Up

We can package all these configurations in a function and call it the main function of the program.

Application #1 – Fixed Duty Cycle

Now let’s utilize this configuration with a simple demo. In this demo, we will set the timing parameters only one time.

This application only consists of 3 main functions:

  • Configure system clock
  • Configure the timer
  • Enable the timer

We can set the same PSC and ARR values 1000 and 50,000 respectively. These parameters will set the period of the PWM signal to be 500ms and the duty cycle is set in the configuration for 50%.

#include "main.h"

int main ( void )
{
	system_clock_config ();
	tim11_config ( 1000, 50000 );

	tim11_enable ( 1 );

	while (1)
		;

	return 0;
}

As a result we get a PWM signal with a constant 50% duty cycle. And notice that we have just configured the timer peripheral and the program is free to do anything while the PWM signal is ongoing. (Ignore the clock rate error of my chip, I have covered this error previously)

Application #2 – Variable Duty Cycle

Now let’s build on top of the first application to periodically alter the duty cycle of the PWM signal. Our aim is to generate a PWM signal that represents a triangle wave. (Sinusoidal signals can be represented too but we need a “look-up” table for that.)

In this application we will change the duty cycle after each PWM period, meaning that after each Update Interrupt. When the triangle wave is ramping up we will increment the duty cycle to represent a larger value, and when the triangle wave is ramping down we will decrement the duty cycle to represent a lower value.

To achieve that we will control the ramping status with the ‘rampingUp’ variable. Also, we will define the step we will be incrementing or decrementing the value of the CCR1 register of the TIM11 to change the duty cycle of the signal.

#include "main.h"

#define DUTY_STEP		200

static uint8_t rampingUp = 1;
...

In the main function, we will configure the system clock and the timer again. Notice that we are utilizing the timer with a lower period time. This way we do not have to long to represent a full triangle wave period.

Also, by setting the CCR1 value to zero we set our initial duty cycle to 0%. After that, we enable the timer and put our main function into the infinite loop. This is so long for the main function operations. The rest of the operations will be handled by the interrupt handler callback function.

...
int main ( void )
{
	system_clock_config ();
	tim11_config ( 1000, 5000 );

	TIM11->CCR1 = 0;

	tim11_enable ( 1 );

	while (1)
		;

	return 0;
}
...

In the interrupt callback function of the TIM11, we first check if this function is called due to the Update Interrupt of the TIM11 (ref-1). This step is highly recommended as the callback function can also be called due to the Capture Compare event of the TIM11. Remember that we have enabled that interrupt as well (although not utilized in this application).

Then, we check the ramping mode (ref-2). If we are ramping up and the CCR1 value did not reach the ARR value we continue by incrementing the CCR1 value by the amount we have defined before (ref-3). If the CCR1 value reached the ARR value we change the ramping mode and decrement the CCR1 value (ref-4). Also, we implement similar steps for the ramping down state as well (ref-5). In the end, we clear the interrupt flag to enter this function block after the next interrupt.

...
void TIM1_TRG_COM_TIM11_IRQHandler ( void )
{
  if ( TIM11->SR & TIM_SR_UIF )         /* ref-1 */
  {
    // Ramping up
    if ( rampingUp )                    /* ref-2 */
    {
      if ( TIM11->CCR1 <= TIM11->ARR )  /* ref-3 */
      {
        TIM11->CCR1 += DUTY_STEP;
      }
      // Reached top, go down
      else                              /* ref-4 */
      {
        rampingUp = 0;
        TIM11->CCR1 -= DUTY_STEP;
      }
    }
    // Ramping down
    else                                /* ref-5 */
    {
      if ( TIM11->CCR1 > 0 )
      {
        TIM11->CCR1 -= DUTY_STEP;
      }
      // Reached bottom, go up
      else
      {
        rampingUp = 1;
        TIM11->CCR1 += DUTY_STEP;
      }
    }
    // Clear update interrupt flag
    TIM11->SR &= ~TIM_SR_UIF;
  }
}

To observe the output, the PC12 pin can be connected to a digital signal analyzer. The generated PWM signal starts with a duty cycle of 0% and increases with each period until it reaches 100%. After that, the duty cycle decreases with each subsequent period. This behavior persists until the device is powered off.

Conclusion

In conclusion, pulse width modulation (PWM) is a widely used technique in electronics and is particularly useful for controlling the speed of motors and the brightness of LEDs. By varying the duty cycle of a periodic digital signal, PWM can produce analog results. In this article, we have explored the basics of PWM, including its applications, circuit design, and programming. We have also provided examples of using PWM in microcontroller programming by bare-metal programming. With its versatility and ease of implementation, PWM will continue to be an essential technique for modern electronics.
